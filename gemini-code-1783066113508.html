01 — Positioning

The repository is the product

ApplyOS is not a landing page generator. The landing page is an artifact — the most visible output of a system whose real value is the structured career knowledge base underneath it, the conversation that selects the right narrative for each opportunity, and the learning loop that improves with every application outcome.

The elegance of the architecture is this: every component is a file. profile.json is your career. content_brief.json is your narrative decision. execution_plan.json is the build specification. applications.json is your outcome history. The repo is the product. Everything else is infrastructure that reads and writes files.

The pitch: ApplyOS is a career operating system that captures a professional's experience as structured knowledge, collaborates with them to choose the strongest narrative for each opportunity, and generates personalised application assets while learning from every outcome. The landing page is merely the most visible artifact. The intelligence is the product.
02 — Principles

What never changes

01
One input per run
The only thing that changes per application is the JD. All context is pre-loaded from the repo. The system reads, not asks.
02
No hallucination tolerance
Every claim on a deployed page traces to profile.json. The validator enforces this. Wrong facts are worse than no page.
03
Tool agnostic
No AI tool is prescribed. Claude, Cursor, Antigravity, direct HTML edit — any tool that can edit a file works in this system.
04
File is the interface
The HTML file is what gets tweaked, watched, versioned, deployed. The file watcher bridges editing and deployment without prescribing a tool.
05
Human at the angle
The brainstorm is not a gate — it is the product. Automation handles deterministic work. The designer decides what story to lead with.
06
Profile compounds
profile.json is a living artifact. One case study today, three tomorrow. Every update makes every future application better without rebuilding anything.
03 — The Brainstorm

This is the product

GPT can generate pages. Claude can write copy. Cursor can edit HTML. None of them do what the brainstorm does: a structured conversation where an AI proposes a specific narrative angle for a specific person's career against a specific role, and the human decides. That interaction is the IP that cannot be replicated by a simpler system.

The brainstorm produces two locked artifacts — content_brief.json (the narrative decision) and later in V3, page_schema.json (the structural decision). Everything downstream of these files is deterministic. Everything upstream is human judgment.

Act 1
Who I am
Emphasis selection: builder, researcher, founding designer, systems thinker, AI-native. One positioning line written for this company specifically.
Act 2
What I've done
2–3 proof of works selected and framed for this domain. FinTech role: KwikPay leads. Builder role: ApplyOS appears. The framing angle, not just the story, is decided here.
Act 3
What I'll bring
15 / 30 / 90 day commitments written for this specific role. Evaluable, specific, credibility-anchored. The section no other application includes.
DS decision
Design system direction
Own DS or company DS? If company DS exists publicly (e.g. Blade, Polaris, Atlassian), the page adopts their visual identity. Decided in the brainstorm, recorded in the brief.
03b — Opportunity Model

Between JD and brainstorm

The brainstorm should reason over a structured understanding of the company — not raw JD text. The Opportunity Model is an intermediate artifact produced automatically after JD analysis, before the brainstorm begins. It gives the brainstorm cleaner input and makes the reasoning reusable across future applications to the same company.

Why this matters: raw JD text is noisy — generic HR language, boilerplate requirements, inconsistent formatting. The Opportunity Model extracts only what matters for narrative decisions: what the company is actually building, how mature their design function is, what they signal about their culture through their word choices, and what differentiated positioning would resonate.

Opportunity Model schema

opportunity_model.json

{
  "company": "Acme Corp",
  "stage": "Series B",                     ← inferred from JD + context
  "design_maturity": "emerging",          ← nascent / emerging / mature / leading
  "product_category": "B2B SaaS",        ← from domain analysis
  "hiring_signal": "first senior IC hire",  ← what the role signals about the team
  "cultural_tone": "builder-first",        ← from JD language analysis
  "public_ds_url": null,                   ← populated if found
  "recommended_positioning": "founding designer operating system",
  "differentiators": [
    "Kuwait relocation = market-first research",
    "SDUI = designer-led architecture",
    "ApplyOS = builder identity in artefact form"
  ]
}

JD URL
  ↓
jd_analysis.json                   ← raw extraction
  ↓

As the profile grows to 40+ case studies, sending everything wastes context and adds noise. The Profile Matcher selects based on the opportunity model's signals before the brainstorm begins.

05 — Design System Layer

Own DS or company DS

Every application has a design system decision made in the brainstorm and recorded in content_brief.json. The pipeline has two paths. The decision is not hardcoded — it is a runtime parameter the script reads from the brief.

brief.design_system: "own"
Script loads design-system/default.md — the designer's visual system written as codegen instructions. Tokens, typography, tone rules, component patterns. Always present in the repo as the fallback.
brief.design_system: "company"
Script reads brief.company_ds_url. At runtime, fetches and extracts tokens (colors, typography, spacing) from the company's public DS. Passes extracted tokens to codegen as a theme layer. No caching — fetched fresh each run. Falls back to default.md if fetch fails or tokens cannot be extracted.
Why company DS matters: consuming Razorpay Blade for a Razorpay application is not a styling choice — it is a signal. The page looks like it was built inside their design system because it was. For design-aware companies, this is the single most differentiating element of the application. No other application tool does this.

Fallback chain

1. Company DS URL provided + tokens extractable    → use company tokens
2. Company DS URL provided + extraction fails       → warn + use default.md
3. No DS URL provided                                → use default.md
4. default.md missing                                 → error: run setup first

DS extraction — one focused LLM call

When a company DS URL is in the brief, generate.js runs a single extraction call before codegen:

System: You are a design token extractor.
        Read the content at this URL.
        Extract: primary colors, background colors, typography
        (font families, sizes, weights), border radius, spacing scale,
        and semantic color names.
        Output as a flat codegen instruction file.
        If you cannot extract tokens, say FALLBACK.

Input: [company_ds_url from brief]
Output: DS tokens → passed directly to codegen call
06 — Confidence Scorer

Explainable, not arbitrary

The confidence scorer produces an explainable result across seven dimensions. The overall score is a weighted average. Thresholds are designer-configurable via .env — not hardcoded. Roles below the skip threshold are never silently discarded: they surface with the reason and an override prompt.

Skill Match
Hard skills listed in JD vs skills in profile.json
Weight 20%
Story Strength
Do the best stories directly address the JD's top signals
Weight 20%
Industry Match
Domain overlap — FinTech, B2B SaaS, Consumer, AdTech
Weight 15%
Leadership Match
Seniority signal — IC vs player-coach vs founding
Weight 15%
Company Stage Fit
Seed vs growth vs public — experience maps to stage
Weight 15%
Domain Experience
Depth in specific product category
Weight 10%
Portfolio Relevance
Published work shows the right kind of output
Weight 5%

Configurable thresholds

CONFIDENCE_AUTO_SKIP=40      ← below: surface with reason + override prompt
CONFIDENCE_AUTO_PROCEED=65  ← above: proceed to brainstorm automatically
A highly crafted application can punch above its weight on a low-scoring role. The auto-skip threshold is a starting point, not a rule. Roles below it always reach the designer with the specific gap flags — never disappear silently.

Actionable output format

The scorer doesn't just output a number. It outputs an actionable breakdown with a specific recommendation for what to lead with.

07b — generate.js Internals

One file, six responsibilities

generate.js is a single script but it has clearly separated internal modules. Even if everything ships in one file, naming the responsibilities matters — it makes future contributors immediately understand what each part does and where to make changes.

generate.js — internal module map
Planner
Reads content_brief.json. Produces execution_plan.json. Writes run.json with initial state. Determines DS path (own or company). The only module that makes decisions.
Knowledge Loader
Reads profile.json. Selects the top relevant stories based on the opportunity model signals. Passes a filtered context to codegen — not the full career history.
DS Loader
If brief.design_system === "company": fetches the URL, runs the extraction LLM call, returns CSS token block. If "own": reads default.md. Falls back to default.md on any failure. Never saves company DS to disk.
Codegen
Calls Claude API with execution_plan + DS tokens + section specs. One call per dynamic section. Returns HTML blocks. Updates run.json stage after each section completes.
Assembler
Loads static section files (nav, footer, whats_this). Injects dynamic HTML blocks into slots. Injects DS token block. Injects PostHog snippet with application event tag. Writes output/[slug]/vN.html and current.html.
Publisher
On --publish: POSTs current.html to Vercel Deploy API. Receives live URL. Writes to applications.json with story performance fields seeded. Generates cold message via one LLM call. Prints URL + cold message. Marks run.json as complete.

CLI modes

generate.js --brainstorm [JD URL]  ← fetch JD + spin up localhost:3000 chat UI (V1.5)
generate.js --run [brief.json]       ← full pipeline: plan → DS → codegen → preview → watch
generate.js --resume                ← read run.json, continue from last completed stage
generate.js --deploy [file.html]    ← deploy only, no API key needed
generate.js --publish               ← promote preview to live, log, print cold message
generate.js --watch                 ← file watcher only, no generation
07c — Failure & Recovery

What happens when things break

Every stage of the pipeline can fail. The system needs a defined response to each failure mode: retry, fallback, or manual intervention. run.json makes resumption possible — a failure at codegen does not require restarting from JD analysis.

StageFailureAuto responseIf auto fails
JD fetchURL blocked, requires login, 404Retry once with a different user-agent. Surface error with the URL.Prompt designer to paste JD text manually. brainstorm.sh accepts text input as fallback.
Opportunity modelLLM call timeout or invalid JSON outputRetry once. Parse partial output if possible.Fall back to raw jd_analysis.json — skip opportunity model, brainstorm reads JD directly.
Company DS fetchURL 404, extraction returns FALLBACK, tokens incompleteLog warning. Automatically fall back to default.md. Never block the pipeline on DS failure.N/A — fallback is always available.
Codegen — sectionAPI timeout, context limit exceeded, invalid HTML outputRetry once with a shorter context. If section fails twice, use a pre-written static fallback for that section type.run.json marks stage as partial. Designer can --resume or replace the section manually.
HTML assemblyMissing section file, slot injection failsLog which section is missing. Skip it and assemble the rest.Print diff of what was skipped. Designer fixes manually before --publish.
Vercel deployAuth error, rate limit, network failureRetry up to 3 times with exponential backoff.Save HTML locally with a timestamp. Print manual deploy instructions. Log as "pending" in applications.json.
Full pipeline crashUnhandled exception at any stageWrite current state to run.json before exiting. Log the exact stage and error.Designer runs --resume. Pipeline reads run.json and continues from last completed stage.

run.json — execution state

{
  "runId": "2026-001",
  "slug": "acme-senior-designer",
  "brief": "briefs/acme-senior-designer.json",
  "stage": "codegen",                          ← current or last attempted
  "status": "failed",                           ← running | completed | failed | partial
  "startedAt": "2026-06-27T14:32:00Z",
  "completedStages": ["plan","ds-load","knowledge-load"],
  "failedStage": "codegen",
  "error": "API timeout after 30s on act2_work section",
  "resumeFrom": "codegen"                      ← generate.js --resume reads this
}
The key principle: no stage reruns a completed stage on resume. If plan and DS load completed, --resume skips directly to codegen. This is how mature pipelines behave.

V3 — Composable Pages 13 — V3

The page as a decision

V3 replaces template selection with template generation. The brainstorm agent produces a page_schema.json in addition to the content brief — a structural decision about which blocks to include, in what order, in what format. The page structure itself is generated per application, not picked from a list.

V3 also introduces layout reasoning: the planner doesn't ask "which template?" — it asks "what narrative structure fits this company, this role, this designer?" An essay-style narrative for a founder. Evidence-first for an engineering culture. Cinematic and minimal for a design-forward company.

Aesthetic risk: as block types multiply and the agent gains structural control, the risk of diluting a curated aesthetic increases. The designer approval gate on page_schema.json is the quality guardrail. No block is added without sign-off. Control over the rendering layer is essential.

14 — Story Performance Memory

The moat

The moat is not the landing page generator. It is the learning system that knows which stories convert for which role types, which design systems resonate with which company cultures, and which narrative angles get more time-on-page. This data accumulates from day one and becomes the primary input for confidence scoring and story selection over time.

# Example: story performance after 30 applications

"story_performance_aggregate": {
  "kwikpay-sdui": {
    "shown": 18, "interviews": 6, "responses": 9, "ghosted": 9,
    "best_for": ["fintech", "founding", "systems-thinker"],
    "ds_with": { "own": 10, "company": 8 }
  },
  "applyos-builder": {
    "shown": 12, "interviews": 8, "responses": 10, "ghosted": 2,

    "ds_with": { "own": 4, "company": 8 }

This data is produced automatically by the analytics agent and read by the confidence scorer and brainstorm agent on every subsequent run. The system learns which stories convert and which design system choices land best — not which stories seem impressive in isolation.

16 — MCP Layer

Chat-native — Tier 3 only

MCP (Model Context Protocol) makes ApplyOS feel native inside Claude chat for Tier 3. The pipeline runs invisibly behind the conversation. The MCP connector is a thin wrapper around the same API endpoints generate.js uses — it does not change the pipeline, it removes the manual steps between stages.

When to build: after people are asking "can I run this entirely inside Claude?" Not before. MCP is a distribution layer, not proof of product-market fit. Nobody buys MCP. People buy "I got interviews."
